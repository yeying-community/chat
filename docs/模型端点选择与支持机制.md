# 模型端点选择与支持机制

本文档是 [AI Native 能力分层架构](./AI-Native能力分层架构.md) 中模型层的运行机制补充，重点说明模型供应商、模型能力和端点选择。

本文用于说明：Chat 在调用大模型接口时，如何判断可用端点、如何选择最终端点，以及排查问题时应从哪一层看。

## 1. 先看结论

- 运行时不会“自动探测后端支持哪些端点”。
- 端点选择由模型元数据和代码规则共同决定：`providerName` 选客户端实现类，`supported_endpoints` 和模型类型决定具体 path。
- 新实现优先使用现代端点：OpenAI 兼容模型优先 `/v1/responses`，Anthropic 模型优先 `/v1/messages`。
- `/v1/chat/completions` 是兼容回退端点，不作为新能力和 Skills 的默认目标。
- 会话运行时可选模型只以 `/v1/models` 返回结果为准。
- 创建面具时看到的候选模型来自 provider models 接口 `/api/v1/public/providers/models`，它的范围可以比 `/v1/models` 更大。
- 一个端点是否真的可用，要看三层交集：

1. 前端常量与客户端实现（想调用什么）
2. Next.js `/api/*` 代理路由（是否允许转发）
3. Router/上游服务（是否真实支持）

## 2. 运行时如何选择端点

### 2.1 请求入口

- 入口在 `app/store/chat.ts` 的 `onUserInput()`。
- 实际调用通过 `getClientApi(providerName)` 获取 provider 对应实现，再执行 `api.llm.chat(...)`。

### 2.2 按 provider 选择实现类

- 映射在 `app/client/api.ts` 的 `getClientApi()`。
- 例如：
  - `OpenAI/Azure -> ChatGPTApi`
  - `Google -> GeminiProApi`
  - `Anthropic -> ClaudeApi`
  - `Volcengine -> VolcengineApi`
  - `DeepSeek -> DeepSeekApi`
  - `ChatGLM -> ChatGLMApi`

### 2.3 按模型进一步选择 path

- OpenAI/Azure（`app/client/platforms/openai.ts`）：
  - 文本和多模态聊天优先使用 `v1/responses`
  - GPT 系列模型同时支持 `v1/responses` 和 `v1/chat/completions` 时强制选择 `v1/responses`
  - 只有模型或 Router 只声明支持旧接口时，才回退到 `v1/chat/completions`
  - `dall-e-3`：`v1/images/generations`
  - Azure：`/deployments/{deploy}/...` 路径
- Anthropic（`app/client/platforms/anthropic.ts`）：
  - 优先使用 `v1/messages`
  - `v1/messages` 也是 Anthropic 工具、多模态和新模型能力的主要承载端点
- Alibaba（`app/constant.ts` + `app/client/platforms/alibaba.ts`）：
  - 模型名包含 `vl/omni` 走多模态端点
  - 否则走文本端点
- Volcengine（`app/constant.ts` + `app/client/platforms/volcengine.ts`）：
  - 当前聊天走 `api/v3/chat/completions`
  - 代理入口为 `/api/volcengine`
- ChatGLM（`app/client/platforms/glm.ts`）：
  - `cogview-*` -> image path
  - `cogvideo-*` -> video path
  - 其他 -> chat path
- Google（`app/client/platforms/google.ts`）：
  - 固定 `:streamGenerateContent`
  - 流式时附加 `alt=sse`

## 3. 如何判断“支持的端点”

必须按下面顺序检查：

### 3.1 前端常量与客户端实现

- 看 `app/client/endpoints.ts` 中 `SupportedTextEndpoint`、`SupportedEndpoint` 和选择函数。
- 看 `app/constant.ts` 中 `OpenaiPath`、各 provider 的 `ChatPath` 定义。
- 看 `app/client/platforms/*.ts` 中 `chat()` 的分支逻辑。
- 文本端点选择入口是 `selectPreferredRequestEndpoint()`：
  - 优先顺序默认是 `/v1/responses` -> `/v1/messages` -> `/v1/chat/completions`
  - GPT 系列模型只要同时支持 `/v1/responses` 和 `/v1/chat/completions`，就使用 `/v1/responses`
  - 图片生成类端点单独判断 `/v1/images/generations`

### 3.2 Next.js 代理路由

- 统一分发入口：`app/api/[provider]/[...path]/route.ts`。
- OpenAI 有 allowlist：`app/api/openai.ts`（只放行 `OpenaiPath` 列表）。
- Anthropic 有 allowlist：`app/api/anthropic.ts`。
- Volcengine 代理入口：`app/api/volcengine.ts`。
- 其他 provider 多数是去掉 `/api/{provider}` 前缀后直接透传。

### 3.3 Router 或上游真实路由

- Router `v1` 路由注册：`internal/transport/http/router/relay.go`。
- 路径到 relay mode 的判定：`internal/relay/relaymode/helper.go`。
- `/api/v1/public/*` 也有一套同类公共 relay 路由。

## 4. 当前仓库的现状特征（与排查强相关）

- 默认 OpenAI 地址是 Router 后端地址（由 `routerBackendUrl` 注入）。
- 设置页当前只提供 OpenAI provider 切换项。
- 前端启动时会并行拉两份数据：
  - 运行时模型列表：`/v1/models`
  - provider models 列表：`/api/v1/public/providers/models`
- 面具里的 `candidateModels` 只定义“允许范围”，不等于会话里一定可选。
- 模型元数据里的 `supported_endpoints` 用于决定运行时请求 path。
- 实际创建会话时，会话可选模型 = `candidateModels ∩ /v1/models`。
- 如果某个面具的候选模型和 `/v1/models` 没有交集，会明确报错并阻止创建会话。
- 如果只有部分候选模型仍存在于 `/v1/models`，则会话中只显示仍可用的那部分。
- 如果面具默认模型已经不在当前运行时列表里，不会自动回退到别的模型，而是要求用户手动切换到可用模型。
- 在 Router + UCAN 模式下，未授权时会跳过 OpenAI `models()` 拉取。
- 当前 OpenAI 兼容聊天正文已支持 `v1/responses` 请求体转换、流式解析和工具调用解析。
- 当前 Anthropic 聊天正文已支持基于 `endpointPath` 选择 `v1/messages`。

## 5. 面向 Skills 的端点原则

- 新技能不要默认面向 `/v1/chat/completions` 设计能力。
- OpenAI 兼容技能应优先声明和使用 `/v1/responses`，用于承载多模态输入、工具调用、结构化输出和后续 agentic 能力。
- Anthropic 技能应优先声明和使用 `/v1/messages`，用于承载多模态输入、工具调用和新模型能力。
- 如果技能依赖某个现代能力，manifest 应通过模型候选项或能力说明约束可选模型，避免运行时静默降级。
- 只有为了兼容旧 provider、旧 Router 或第三方 OpenAI-compatible 服务时，才允许回退到 `/v1/chat/completions`。

## 6. 建议的排查顺序

1. 先确认当前会话 `providerName` 与 `model`。
2. 在 `/api/v1/public/providers/models` 返回数据里确认该模型的 `supported_endpoints`。
3. 在 `app/client/endpoints.ts` 看端点选择结果。
4. 在对应 `platforms/*.ts` 看 `chatPath` 实际构造逻辑。
5. 确认当前 base URL（是 `/api/*`、Router 地址，还是直连上游地址）。
6. 若走 `/api/*`，检查对应 `app/api/*.ts` 是否允许该子路径。
7. 若走 Router，检查 Router 路由与 relay mode 映射是否覆盖该路径。
8. 最后再看上游 channel/模型本身是否支持该 API 形态。
