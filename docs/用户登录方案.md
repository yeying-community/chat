# 用户登录方案

> 本文档基于当前代码实现重写，统一收口登录页、钱包 UCAN、中心化 UCAN、Router / WebDAV 鉴权边界，以及移动端差异。

## 1. 文档范围

本文档回答的是“当前系统实际上怎么登录和授权”，不是未来方案评审。内容以现有代码为准，重点覆盖：

1. 当前有哪些可用的登录 / 授权方式。
2. `/auth` 页面现在承担什么角色。
3. 钱包 UCAN、中心化 UCAN、Access Code / API Key、WebDAV Basic Auth 分别作用在哪。
4. Router 与 WebDAV 为什么能在 UCAN 模式下一次授权、多后端访问。
5. 本地到底存了哪些授权状态。
6. 什么情况下只是需要解锁钱包，什么情况下必须重新授权。
7. 移动端为什么不能直接复用 PC 端的钱包体验。

## 2. 当前结论

先说结论，避免把“登录”和“请求鉴权”混在一起：

1. 当前没有单一统一登录体系，而是多套能力并存。
2. `/auth` 已经是当前登录入口，但它不是一个带全局路由守卫的 SSO 页面。
3. 钱包 UCAN 和中心化 UCAN，都是跨 Router / WebDAV 的授权方式。
4. Access Code / API Key 主要用于 Router / 模型请求，不等于 WebDAV 登录。
5. WebDAV Basic Auth 只作用于 WebDAV 同步，不等于 Router 登录。
6. 当前移动端如果没有钱包插件，实际可用路径是中心化 UCAN，或者继续分别配置 Router 凭证与 WebDAV Basic 凭证。

## 3. 当前支持的登录 / 授权方式

| 方式 | 入口 | 当前作用范围 | 是否跨 Router / WebDAV |
|---|---|---|---|
| 钱包 UCAN | `/auth` 页点击登录，调用 `connectWallet()` | Router + WebDAV | 是 |
| 中心化 UCAN | `/auth` 页输入地址后跳转中心化授权 | Router + WebDAV | 是 |
| Access Code / API Key | `/auth` 页 / 设置页 | 主要是 Router / 模型请求 | 否 |
| WebDAV Basic Auth | 同步设置页 | 仅 WebDAV | 否 |

代码落点：

1. 登录页：`app/components/auth.tsx`
2. 钱包 UCAN：`app/plugins/wallet.ts`
3. 中心化 UCAN：`app/plugins/central-ucan.ts`
4. Router 鉴权兜底：`app/store/access.ts`
5. WebDAV 直连 / 代理：`app/plugins/webdav.ts`、`app/utils/cloud/webdav.ts`、`app/api/webdav/[...path]/route.ts`

## 4. `/auth` 页面当前职责

当前 `/auth` 页已经不只是“填 Access Code”的老入口，而是实际登录入口，主要做 3 件事：

1. 展示当前 UCAN 授权状态：`checking / authorized / expired / unauthorized`
2. 支持钱包登录
3. 支持中心化 UCAN 登录

关键逻辑在 `app/components/auth.tsx`：

1. 页面加载时调用 `isValidUcanAuthorization()` 判断是否已授权。
2. 如果 URL 上带有中心化授权回调 `code`，会自动执行 `exchangeCentralAuthorizeCode()`。
3. 主登录按钮 `handlePrimaryLogin()` 按环境变量决定优先走哪条链路：
   - `UCAN_LOGIN_FORCE_MODE=auto`
   - `UCAN_LOGIN_FORCE_MODE=wallet`
   - `UCAN_LOGIN_FORCE_MODE=central`
4. `wallet` 失败且模式为 `auto` 时，会回退到中心化 UCAN。

### 4.1 当前没有全局 Auth Gate

这里要说清一个边界：

1. 当前项目已经有登录页。
2. 但当前没有实现“所有页面先过统一路由守卫，再决定是否跳 `/auth`”的全局 Auth Gate。
3. 也就是说，之前评估过的那套路由守卫式前置登录，目前还是评估，不是现状实现。

这部分旧文档里最有价值的信息，就是这个边界本身，所以保留在这里，不再单独留一份评估文档。

## 5. 系统如何判断“已登录”

当前“是否已授权”不是单看一个 token，而是按不同模式分别判断。

### 5.1 UI 层 UCAN 授权判断

`app/plugins/wallet.ts` 的 `isValidUcanAuthorization()`：

1. 如果当前是中心化模式，调用 `isCentralUcanAuthorized()`
2. 否则读取本地 Root，检查：
   - Root 是否存在
   - `exp` 是否未过期
   - `iss` 是否和当前钱包地址匹配
   - capabilities 是否匹配当前预期
   - `service_hosts` 是否匹配当前 Router / WebDAV host

### 5.2 Router 请求级“可用性”判断

`app/store/access.ts` 的 `isAuthorized()` 额外把以下情况也视为“当前可以工作”：

1. 用户已经填了某个 provider 的 API Key
2. 用户填了 Access Code
3. 服务端 `needCode=false`
4. 当前 Router URL 指向 router 后端，且本地 UCAN 元信息有效，或中心化 UCAN 有效

所以：

1. “已登录”更像是“当前请求有一条可用鉴权路径”。
2. 它不等于“所有后端都已经统一授权成功”。

## 6. 钱包 UCAN 模式

### 6.1 当前模型

钱包 UCAN 不是“拿一个长期 JWT”，而是三层结构：

1. 本地 UCAN session
2. Root UCAN
3. 请求级 Invocation UCAN

### 6.2 本地 UCAN session

`app/plugins/ucan-session.ts` 当前不再依赖请求路径时唤醒钱包，而是本地维护可签名 session：

1. session id 固定为 `default`
2. 优先从 IndexedDB `yeying-web3 / ucan-sessions` 读取
3. 若不存在或已过期，则创建本地 Ed25519 session
4. 请求发起时优先复用本地 session，不主动唤醒钱包

这也是现在“请求时尽量不打断用户”的关键实现。

### 6.3 Root UCAN

钱包登录在 `loginWithUcan()` 中完成，核心步骤：

1. 连接钱包，确定 `currentAccount`
2. 清理中心化模式残留状态
3. 复用现有 Root；如果 Root 过期、地址不匹配、能力不匹配，先清理
4. 生成当前 Root 所需 capabilities
5. 构造 SIWE statement：`buildUcanRootStatement()`
6. 调用 SDK `createRootUcan()` 创建 Root
7. 校验 Root 已写入本地，再写入本地元信息

当前 Root 能力模型在 `app/plugins/ucan.ts`：

1. 资源统一使用 `app:all:<appId>`
2. Router action：`invoke`
3. WebDAV action：`write`
4. Root statement 里会附带：
   - `aud`
   - `cap`
   - `service_hosts.router`
   - `service_hosts.webdav`

### 6.4 Invocation UCAN

请求真正发出去时，不直接拿 Root 当 Bearer token，而是按目标后端临时签发 Invocation：

1. Router 请求链路：`app/client/platforms/router.ts`、`openai.ts`、`anthropic.ts`
2. WebDAV 请求链路：`app/plugins/webdav.ts`、`app/utils/cloud/webdav.ts`
3. 每次都会按目标后端解析：
   - `audience`
   - `capabilities`
4. 再基于本地 session 创建短期 Invocation UCAN

所以表面上看是“一次钱包登录后两边都能用”，本质上其实是：

1. Root 共用
2. Invocation 按后端分别签发

## 7. 中心化 UCAN 模式

中心化 UCAN 是当前无钱包插件场景的正式路径，不是临时兼容。

### 7.1 登录入口

`app/components/auth.tsx` 中：

1. 用户输入或选择区块链地址
2. 前端调用 `createCentralAuthorizeRequest()`
3. 跳转到服务端返回的 `verifyUrl`
4. 完成 TOTP / 服务侧认证后回跳
5. 前端用 `exchangeCentralAuthorizeCode()` 兑换授权结果
6. `applyCentralAuthorizeExchange()` 把结果写入本地

### 7.2 本地持久化内容

`app/plugins/central-ucan.ts` 当前会维护几类状态：

1. `ucanAuthMode=central`
2. `centralAccessToken` / `centralAccessExpiresAt`
3. `centralAuthSubject`
4. `centralIssueSessionToken` / `centralIssueSessionExpiresAt`
5. `centralUcanTokensV1`

其中 `centralUcanTokensV1` 不是单 token，而是按 `audience + capabilities` 缓存的 token 池。

### 7.3 请求时怎么发 UCAN

中心化模式下，请求不会复用“一个全能 token”：

1. 先看本地 `centralUcanTokensV1` 里有没有匹配当前 `audience + capabilities` 的 token
2. 没有就先向服务端申请 issue session：
   - `POST /api/v1/public/auth/central/session`
3. 再按目标后端申请对应 UCAN：
   - `POST /api/v1/public/auth/central/issue`
4. 把签发结果缓存到本地 token 池

这就是为什么当前中心化 UCAN 已经能正确支持 Router + WebDAV 双后端，而不会再把 Router token 误拿去调 WebDAV。

## 8. Router 与 WebDAV 的鉴权边界

这里保留的是之前 Router / WebDAV 集成专题里仍然有效的实现边界。

### 8.1 Router

Router 是模型调用后端，当前特点：

1. 前端通常直连 Router
2. 如果 URL 命中 Router 后端，优先尝试附带 UCAN
3. 若用户手动填了 API Key / Access Code，可以作为兜底 Authorization
4. 钱包模式和中心化模式都会按 Router 的 `audience + capabilities` 单独处理

### 8.2 WebDAV

WebDAV 既有配额接口，也有同步文件接口：

1. 配额接口：`app/plugins/webdav.ts`
   - 浏览器直连 `/api/v1/public/webdav/quota`
   - 需要 WebDAV 服务端正确配置 CORS
2. 同步接口：`app/utils/cloud/webdav.ts`
   - 可直连
   - 也可走 `/api/webdav/*` 代理

### 8.3 WebDAV 代理边界

`app/api/webdav/[...path]/route.ts` 当前做了这些限制：

1. endpoint 必须命中允许列表，避免 SSRF
2. 非 public API 只能访问 `STORAGE_KEY` 目录内路径
3. 禁止路径穿越
4. 非 public API 只允许 `MKCOL / GET / PUT / DELETE`
5. 禁止通过代理删除根同步目录

### 8.4 为什么 UCAN 能跨两个后端

因为当前能力模型和 token 签发都是按目标后端拆开的：

1. Router：`app:all:<appId> + invoke`
2. WebDAV：`app:all:<appId> + write`
3. `audience` 也分别取 Router host 与 WebDAV host
4. Root statement 通过 `service_hosts` 明确声明服务目标

## 9. Access Code / API Key 与 WebDAV Basic Auth

这块很容易让用户误解，必须单列出来。

### 9.1 Access Code / API Key

这条链路来自老的模型访问方式：

1. 服务端接口校验在 `app/api/auth.ts`
2. `Authorization` 里如果不是 `nk-` 前缀，就按 API Key 处理
3. 如果是 `nk-<accessCode>`，按 Access Code 处理
4. 通过后，服务端可注入系统 API Key

它的本质是：

1. 模型请求准入控制
2. 与 UCAN 可以并存
3. 不等于 WebDAV 登录

### 9.2 WebDAV Basic Auth

WebDAV Basic Auth 只用于同步后端：

1. 用户在同步配置里填 WebDAV 用户名 / 密码
2. 它不参与 Router 鉴权
3. 它也不会让 `/auth` 页面自动变成“全局统一登录”

结论：

1. Access Code / API Key 和 WebDAV Basic Auth 是两套边界
2. 当前没有“同一套用户名密码同时登录 Router + WebDAV”的实现

## 10. 本地存储与状态分层

### 10.1 内存缓存

`app/plugins/ucan-session.ts` 维护：

1. `cachedSession`
2. `cachedAt`
3. `sessionPromise`

这层只在当前页面进程有效。

### 10.2 localStorage

钱包模式会写：

1. `currentAccount`
2. `ucanRootExp`
3. `ucanRootIss`
4. `ucanRootCaps`

中心化模式会额外写：

1. `ucanAuthMode`
2. `centralAccessToken`
3. `centralAccessExpiresAt`
4. `centralAuthSubject`
5. `centralIssueSessionToken`
6. `centralIssueSessionExpiresAt`
7. `centralUcanTokensV1`

### 10.3 IndexedDB

当前 UCAN session 相关记录会写到：

1. DB：`yeying-web3`
2. Store：`ucan-sessions`

### 10.4 钱包插件状态

还有一层不在前端本地可完全替代：

1. 钱包连接状态
2. 钱包账户切换
3. 钱包签名能力

因此“浏览器里看起来还有地址”并不一定意味着请求一定还能继续签名。

## 11. 什么时候只是解锁钱包，什么时候必须重新授权

### 11.1 只需要解锁钱包

典型情况：

1. 钱包暂时锁定，但本地 Root 仍有效
2. 当前账户没变
3. `service_hosts` 没变
4. capabilities 没变

这种情况下，现有实现会尽量保留已授权状态，不主动清掉 Root。

### 11.2 必须重新授权

以下情况会导致授权失效或被主动清理：

1. Root 过期
2. 切换了钱包账户
3. Root 的 issuer 和当前账户不匹配
4. Root capabilities 与当前预期不匹配
5. `service_hosts` 与当前 Router / WebDAV host 不匹配
6. 请求路径中出现这些错误：
   - `missing ucan proof chain`
   - `ucan root audience mismatch`
   - `missing ucan session key`
   - `ucan root expired`
   - `ucan root capability mismatch`

这部分错误清理逻辑在 `app/plugins/ucan-auth.ts` 和 `app/plugins/wallet.ts`。

## 12. 移动端与桌面端差异

移动端登录专题里最有价值的结论，当前仍然成立：

1. 移动端问题本质不是“没有登录入口”
2. 而是“没有钱包插件时，不能直接复用 PC 端钱包链路”

当前实际结论：

1. 如果移动端没有钱包插件，优先走中心化 UCAN
2. 如果不用中心化 UCAN，那就只能继续分开处理：
   - Router：Access Code / API Key
   - WebDAV：Basic Auth
3. 这也是为什么“用户名密码统一登录 Router + WebDAV”在当前架构里并不存在

## 13. 当前边界与未实现项

基于当前代码，以下几点要明确：

1. 已实现登录页，但未实现全局路由守卫式前置登录。
2. 已实现钱包 UCAN 与中心化 UCAN 双路径，并且两者都支持 Router + WebDAV。
3. 已实现按 `audience + capabilities` 缓存和签发多后端 token。
4. 未实现统一账号密码同时打通 Router 与 WebDAV。
5. 未实现真正意义上的 SSO / 统一会话中心。
