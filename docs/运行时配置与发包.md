# 运行时配置与发包

本文档说明当前项目中 `.env`、构建产物、运行时公开配置三者的边界，以及发包后的配置修改方式。

## 1. 结论

当前项目已将配置拆为三类：

1. **构建期配置**
   - 在 `next build` / `tauri build` 时确定
   - 改动后需要重新打包
2. **运行期服务端配置**
   - 在服务启动时从 `.env` 读取
   - 改动后重启服务生效
3. **运行期前端公开配置**
   - 由服务端启动后按当前环境变量生成
   - 通过页面 `meta[name="runtime-config"]` 和 `/api/config` 下发
   - 改动后重启服务并刷新页面生效

这意味着：

- **发包后可以修改 `.env`**
- 但只对“运行期配置”生效
- **真正影响构建产物形态的配置仍然需要重新打包**

## 2. 三类配置的边界

### 2.1 构建期配置

当前主要由 `app/config/build.ts` 提供，只保留构建相关元信息：

- `buildMode`
- `isApp`
- `version`
- `commitDate`
- `commitHash`

以及构建流程自身使用的环境变量：

- `BUILD_MODE`
- `BUILD_APP`
- `DISABLE_CHUNK`
- Tauri 签名相关变量

这类变量改变后，必须重新执行打包。

### 2.2 运行期服务端配置

由 `app/config/server.ts` 在请求处理/服务运行时读取，包括：

- 各模型服务的 API Key / URL
- `ENABLE_MCP`
- `CUSTOM_MODELS`
- `DEFAULT_MODEL`
- `VISION_MODELS`
- `HIDE_USER_API_KEY`
- `ENABLE_BALANCE_QUERY`
- `DISABLE_FAST_LINK`

这类变量在**修改 `.env` 并重启服务**后生效。

### 2.3 运行期前端公开配置

由 `app/config/runtime.ts` 统一组装，包括：

- `routerBackendUrl`
- `webdavBackendBaseUrl`
- `webdavBackendPrefix`
- `webdavBackendUrl`
- `centralUcanAuthBaseUrl`
- `centralUcanAppId`
- `ucanLoginForceMode`
- `template`
- `needCode`
- `hideUserApiKey`
- `disableGPT4`
- `hideBalanceQuery`
- `disableFastLink`
- `customModels`
- `defaultModel`
- `visionModels`
- `enableMcp`

这些值会：

1. 在服务端渲染页面时注入到 `meta[name="runtime-config"]`
2. 通过 `/api/config` 返回给前端

所以这类配置在**修改 `.env` 后重启服务，并刷新页面**即可更新。

## 3. 当前推荐部署方式

### 3.1 打包

使用现有发包流程：

- `scripts/package.sh`

该流程会：

- 生成 standalone 产物
- 打包 `scripts/starter.sh`
- 带上 `.env.template`

### 3.2 部署

部署后：

1. 从 `.env.template` 复制出 `.env`
2. 按环境修改 `.env`
3. 使用 `scripts/starter.sh` 启动

`scripts/starter.sh` 会在启动 `server.js` 前显式加载 `.env`。

### 3.3 修改配置

如果只是修改运行期参数，例如：

- `ROUTER_BACKEND_URL`
- `WEBDAV_BACKEND_BASE_URL`
- `CENTRAL_UCAN_AUTH_BASE_URL`
- `DEFAULT_MODEL`
- `CUSTOM_MODELS`
- `ENABLE_MCP`

则流程是：

1. 修改 `.env`
2. 重启服务
3. 刷新页面

不需要重新打包。

## 4. 什么时候必须重新打包

以下场景仍然需要重新打包：

- 切换 `BUILD_MODE`
- 切换是否桌面导出构建 `BUILD_APP`
- 修改 webpack / Next 输出相关行为
- 修改 Tauri 打包参数或签名参数
- 变更依赖、代码、静态资源

## 5. 设计原则

当前项目遵循以下原则：

- `.env` 不再同时承担“构建期配置”和“运行期公开配置注入”的双重职责
- 前端需要的非敏感配置，统一从运行期配置源下发
- Secret 只保留在服务端，不写入 `runtime-config` 或 `/api/config`

## 6. 已知事项

### 6.1 export 模式限制

`BUILD_MODE=export` 仍受 Next 静态导出限制影响。  
当前仓库里某些 API 路由并不满足静态导出要求，例如 `/api/upstash/[action]/[...key]`，这属于既有导出约束，不是运行期配置拆分引入的问题。

### 6.2 JSON import 警告

在 Node 24 + Jest 环境下，`app/config/build.ts` 对 `src-tauri/tauri.conf.json` 的普通 JSON import 会出现弃用警告。  
当前不影响 `build` 和 `test` 通过，但后续若升级相关工具链，建议统一收敛这部分 JSON 读取方式。
