# 运行时配置与发包

本文档说明当前项目里配置的真实边界：哪些变量在打包时生效，哪些在服务启动时生效，哪些会继续下发到前端。

当前模板文件分为两份：

- `.env.template`：运行期模板
- `.env.build.template`：构建/发包模板

推荐配套文件：

- `.env`：实际运行期配置
- `.env.build`：实际构建期配置

## 1. 结论

当前项目已经把配置拆成三层：

1. 构建期配置
   - 在 `next build` / `npm run export` / 桌面端打包时读取
   - 变更后必须重新构建，桌面端场景下通常还要重新发包
2. 服务端运行期配置
   - 在 Node 服务启动时从 `.env` 读取
   - 变更后需要重启服务
3. 前端公开运行期配置
   - 仍然由服务端在启动后从 `.env` 读取
   - 通过 `meta[name="runtime-config"]` 和 `/api/config` 下发给浏览器
   - 变更后需要重启服务，并刷新页面

布尔开关变量当前统一支持：

- `0` / `1`
- `false` / `true`

推荐统一使用 `0` / `1`。

因此现在的标准认识是：

- `.env` 不再等价于“全是构建期变量”
- standalone 部署产物在发包后仍然可以修改 `.env`
- 只要不是构建产物形态相关的变量，就不需要重新打包

## 2. 配置来源

### 2.1 构建期

构建元信息由 [app/config/build.ts](/Users/liuxin2/Workspace/opensource/chat/app/config/build.ts) 提供：

- `version`
- `commitDate`
- `commitHash`
- `buildMode`
- `isApp`

这些字段会被合并进前端公开配置，但它们的来源仍然是构建期。

构建相关模板变量现在放在 `.env.build.template`，实际构建时建议复制为 `.env.build` 使用。  
其中 `BUILD_MODE`、`BUILD_APP` 仍然属于构建期概念，但不再作为 `.env.build` 的手工配置项，而是由 `scripts/package.sh` 的 mode 或专用构建脚本决定。

### 2.2 服务端运行期

服务端私有配置由 [app/config/server.ts](/Users/liuxin2/Workspace/opensource/chat/app/config/server.ts) 读取。

这部分包括：

- 各模型供应商的 API Key / URL
- 访问控制与代理
- MCP 开关
- WebDAV 服务端代理配置
- 分析与基础设施配置

运行期模板变量现在放在 `.env.template`。

### 2.3 前端公开运行期

前端公开配置由 [app/config/runtime.ts](/Users/liuxin2/Workspace/opensource/chat/app/config/runtime.ts) 组装，在服务端渲染时注入：

- 页面首屏通过 `meta[name="runtime-config"]` 读取
- 客户端后续也可以通过 `/api/config` 获取最新公开配置

相关入口：

- [app/layout.tsx](/Users/liuxin2/Workspace/opensource/chat/app/layout.tsx)
- [app/api/config/route.ts](/Users/liuxin2/Workspace/opensource/chat/app/api/config/route.ts)
- [app/config/client.ts](/Users/liuxin2/Workspace/opensource/chat/app/config/client.ts)

## 4. 发包后的正确操作

### 4.1 package.sh 模式

当前 [scripts/package.sh](/Users/liuxin2/Workspace/opensource/chat/scripts/package.sh) 已经是统一打包入口，支持以下模式：

- `standalone`：Node 服务部署包
- `export`：静态导出包
- `app`：桌面端构建产物包
- `app-release`：桌面端 updater release 产物包

这里的 `mode` 是以下构建期行为的唯一来源：

- `BUILD_MODE`
- `BUILD_APP`

常用参数：

- `--output-dir DIR`：指定打包产物输出目录，默认是仓库下的 `output/`

示例：

- `bash scripts/package.sh`
- `bash scripts/package.sh standalone`
- `bash scripts/package.sh export v1.2.3`
- `bash scripts/package.sh app`
- `bash scripts/package.sh app-release`
- `bash scripts/package.sh standalone --output-dir ./dist`

### 4.2 standalone 部署

推荐流程：

1. 通过现有发包流程生成 standalone 产物
2. 构建前，如需切换构建细节变量，先从 `.env.build.template` 复制出 `.env.build`
3. 按构建目标修改 `.env.build`
4. 部署时，从 `.env.template` 复制出 `.env`
5. 按部署环境填写 `.env`
6. 启动服务

后续如果只是修改运行期变量：

1. 更新 `.env`
2. 重启服务
3. 刷新页面

不需要重新打包。

### 4.3 standalone 打包流程

standalone 模式推荐流程：

1. 构建前，如需切换构建细节变量，先从 `.env.build.template` 复制出 `.env.build`
2. 按构建目标修改 `.env.build`
3. 部署时，从 `.env.template` 复制出 `.env`
4. 按部署环境填写 `.env`
5. 执行 `bash scripts/package.sh standalone`
6. 在产物目录中使用 `scripts/starter.sh` 启动服务

### 4.4 什么时候必须重新打包

以下情况仍然必须重新构建，桌面端场景通常还要重新发包：

- 切换 `scripts/package.sh` 的 mode
- 修改 `DISABLE_CHUNK`
- 修改 Tauri 签名参数
- 变更代码、依赖、静态资源
- 任何影响 Next / Tauri 产物形态的配置

## 5. 当前实现的设计原则

当前实现遵循下面几条边界：

1. `build.ts` 只保留真正的构建元信息
2. `server.ts` 负责服务端私有配置
3. `runtime.ts` 负责浏览器可见的运行时公开配置
4. Secret 不进入 `runtime-config`，也不进入 `/api/config`
5. 前端不再依赖构建时固化的公开配置来决定运行期行为

## 6. 排障方法

### 6.1 看变量属于哪一层

先判断问题变量属于哪类：

- build: 重建才会变
- server runtime: 重启服务才会变
- public runtime: 重启服务后还需要浏览器刷新

### 6.2 验证前端实际拿到的公开配置

可以检查：

- 页面里的 `meta[name="runtime-config"]`
- `GET /api/config` 返回值

如果两者都没更新，优先检查：

1. 服务是否已经重启
2. 启动脚本是否正确加载了新的 `.env`
3. 页面是否仍在使用旧标签页缓存

## 7. 已知事项

### 7.1 export 模式限制

`BUILD_MODE=export` 仍受 Next 静态导出约束影响。  
仓库里某些 API 路由本身不适合静态导出，这属于既有约束，不是这次配置拆分引入的问题。

### 7.2 Jest / Node 24 的 JSON import 警告

`app/config/build.ts` 当前直接 import 了 `src-tauri/tauri.conf.json`。  
在 Node 24 + Jest 环境下会出现 JSON import 的弃用警告，但当前不影响 `build`、`test` 和 `export` 通过。
