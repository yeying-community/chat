# 问题排查（打样版）

本文档用于沉淀一套可复用的故障排查方法，并完整记录一次真实案例，作为后续同类问题的依据与模板。

## 1. 适用范围

- Chat 调用 OpenAI 兼容接口时出现的请求失败、响应异常、能力不符合预期等问题。
- 尤其适用于 `503 service_unavailable_error` 这类“表面看像上游不可用，但实际可能是请求不合规”的场景。

## 2. 记录规范（必须项）

每次排查至少记录以下内容：

- 现象：用户看到的报错原文、出现条件（文本/图片/PDF、单轮/多轮、是否流式）。
- 环境：时间窗口、仓库分支、关键配置（模型、端点、协议类型）。
- 证据：至少 2 组 `trace id`（失败 + 成功）与对应日志位置。
- 排除项：哪些假设已被证据否定（例如 UA、模型能力、渠道状态）。
- 根因：必须落到“请求字段级别/代码逻辑级别”，不能只写“上游不稳定”。
- 修复：修改点（文件/函数/字段映射规则）与变更目的。
- 验证：修复前后对照结果（同条件复测）。
- 回归项：后续需要补的测试或监控。

## 3. 标准排查流程

1. 固定时间窗口，先抓“同一窗口内的失败与成功样本”。
2. 以 `trace id` 为主键串联 `relay.log / router.log / api.log`，避免凭感觉推断。
3. 明确分层：客户端构造 -> Router 透传/转换 -> 上游响应。
4. 用最小变量集做 A/B：同 token、同渠道、同模型、同图片，只改一个因素。
5. 先排除“非结构因素”：网络、渠道离线、配额、鉴权。
6. 再比对请求体差异：`role`、`content.type`、字段名、是否带历史错误文本。
7. 将“失败请求体”和“成功请求体”并排写入结论。
8. 修复后复跑同一条链路，保留新 trace 作为验收证据。

## 4. 案例 001：`/v1/responses` 图片多轮对话间歇性 503

### 4.1 现象

- Chat 在 `responses` 模式下发送图片对话，用户侧收到：
  - `当前分组可用上游暂时不可用，请稍后再试`
- 但同渠道下 Codex CLI 经常可成功，用户怀疑不是上游真实不可用。

### 4.2 关键时间与样本

- 失败样本（用户现场）：
  - `2026041608140724428100025123324`
- 成功样本（Codex，单次请求成功）：
  - `2026041610274141943900095365169`

对应日志可在 Router 仓库中定位：

- `router/logs/relay.log`
- `router/logs/router.log`
- `router/logs/api.log`

### 4.3 排查与证据链

#### A. 先排除“UA 导致被控”

- 同一渠道、同一 token，使用自定义 UA `chat-local-ab/1.0` 进行本地复测。
- 结论：存在“同 UA 既可成功也可失败”的情况，UA 不是决定性因素。

#### B. 再排除“模型不支持图片”

- 单轮图片请求在同渠道下成功：
  - `gpt-5.3-codex` 成功 trace：`2026041610461793352400063015021`
  - `gpt-5.4` 成功 trace：`2026041610463318901700021219517`
- 结论：模型侧图片能力可用，不是“模型不支持图片”。

#### C. 锁定“请求体结构差异”

失败请求（修复前）在 `upstream_request_body` 中可见：

- `assistant` 历史消息被编码为 `type: "input_text"`。
- 多轮中带图片时，上游返回统一 `503 service_unavailable_error`。
- 典型失败 trace：
  - `2026041610473152833300051706517`
  - `2026041610480462918600041464513`

成功请求（修复后）在 `upstream_request_body` 中可见：

- `assistant` 历史消息编码为 `type: "output_text"`。
- 同类多轮 + 图片请求恢复 200。
- 典型成功 trace：
  - `2026041610505774939200068916137`

### 4.4 根因（字段级）

根因是客户端 `responses` 输入归一化逻辑对 `assistant` 角色处理不正确：

- 旧逻辑将文本统一转换为 `input_text`，未按 `role` 区分。
- 在部分上游实现中，`assistant` 历史消息应使用 `output_text`。
- 导致“多轮 + 图片 + 历史 assistant 文本”组合触发上游拒绝，外层表现为 503。

### 4.5 修复方案

修复文件：

- `app/client/platforms/openai.ts`
- `app/utils.ts`
- `app/store/chat.ts`

修复点：

- 新增按角色映射文本类型：
  - `assistant -> output_text`
  - 其他角色 -> `input_text`
- 在 `normalizeResponsesInputMessages` 中传入 `role` 进行归一化。
- 兼容已有 `text / input_text / output_text` 的统一处理。
- 标题生成增加语言脚本清洗：
  - 中文环境（`cn/tw`）仅保留中文、英文字母、数字与空格。
  - 日文环境（`jp`）保留汉字、平假名、片假名、英文字母、数字与空格。
  - 韩文环境（`ko`）保留韩文、汉字、英文字母、数字与空格。
- 标题落库前增加兜底：
  - 清洗后为空时回退 `默认标题`，避免出现空标题。

核心函数：

- `getResponsesTextContentType`
- `normalizeResponsesInputContent`
- `normalizeResponsesInputMessages`
- `trimTopic`

### 4.6 验证结果

- TypeScript 检查通过：
  - `npx tsc -p tsconfig.json --noEmit`
- 实测对照：
  - 修复前：多轮图片 + assistant `input_text` -> 503
  - 修复后：多轮图片 + assistant `output_text` -> 200
- 标题鲁棒性：
  - 输入 `问题排查 დახმარება` 清洗后为 `问题排查`（去除非目标脚本字符）。

### 4.7 回归建议

- 增加单元测试（建议）：
  - `responses` 归一化时不同 `role` 的 `content.type` 映射测试。
  - 覆盖 `text / input_text / output_text` 三种输入形式。
- 增加联调用例（建议）：
  - 单轮文本
  - 单轮图片
  - 多轮图片 + 历史 assistant 文本
  - 历史中包含“错误 JSON 文本”的继续对话
- 日志增强（建议）：
  - 在 debug 日志中保留简化后的 `role + content.type` 结构摘要，便于秒级比对。
