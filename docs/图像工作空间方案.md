# 图像工作空间方案

本文用于沉淀 Chat 应用中图像工作空间（Image Studio）的产品定义、技术选型与第一期范围，作为后续持续讨论与实现的唯一主文档。

## 1. 文档范围

本文统一覆盖以下内容：

- 为什么需要图像工作空间
- 图像工作空间与普通对话会话的区别
- 会话结构与产品命名
- 图像模型策略
- 开源方案选型结论
- 第一期 PRD
- 后续演进路线

从现在开始，图像能力相关讨论优先维护本文，不再并行维护多份重复设计稿。

## 2. 产品定义

目标不是做一个“带生图按钮的普通 chat”，而是做一个新的工作空间类型。

第一版产品定义如下：

- 产品名：`Image Studio`
- 会话类型：`studio`
- 工作空间领域：`image`

建议会话结构：

```ts
type SessionType = "chat" | "studio";

type StudioKind = "image";

type ChatSession = {
  id: string;
  topic: string;
  type?: SessionType;
  studio?: {
    kind: StudioKind;
  };
};
```

这样定义的原因：

- `image` 作为一级会话类型太宽，后续容易膨胀。
- `studio` 更准确表达“工作台式会话形态”。
- `kind = image` 让后续扩展其他 studio 时不需要回头改结构。

## 3. 当前现状

当前仓库已经具备以下基础：

- 会话可继承默认模型、参数和提示词
- provider models 与运行时模型已拆分
  - 创建候选模型来源：`/api/v1/public/providers/models`
  - 会话运行时模型来源：`/v1/models`
- OpenAI 路径已支持 `/v1/images/generations`
- 会话里已有上传图片、尺寸、质量、风格等部分 UI

当前主要不足：

- 图像能力仍然附着在普通 chat 上
- 参数和判断逻辑仍偏 `dall-e-3`
- OpenAI 图像分支目前更像“文生图能力”，不是完整图像工作流
- 会话 UI 仍是消息流主导，不是画布与操作主导

## 4. 产品边界

Image Studio 不是完整 Photoshop，也不是单次 prompt 工具。

第一版目标：

- 用户可以一键进入图像工作空间
- 用户可以围绕一张图持续进行多轮创作
- 会话的主交互区域从消息流转为画布和参数面板

第一版不追求：

- Photoshop 式图层系统
- 完整局部重绘系统
- 多工作区
- 图像资产库
- 批量工作流编排

## 5. 模型策略

产品层不应绑死在某一个固定模型名上。

即使当前默认模型是 `gpt-image-2`，实现也应该基于能力判断，而不是基于模型名硬编码。

前端应优先依据：

- 是否支持 `/v1/images/generations`
- 是否支持 `/v1/images/edits`
- 是否支持多图输入
- 是否支持透明背景
- 是否支持局部重绘 / mask
- 是否支持扩图 / 变体

关于 `gpt-image-2`：

- 截至 `2026-05-17`，OpenAI 官方公开文档中明确可见的 GPT Image 模型为：
  - `gpt-image-1.5`
  - `gpt-image-1`
  - `gpt-image-1-mini`
- 当前没有看到 `gpt-image-2` 作为官方公开型号名
- 这不影响 router 暴露 `gpt-image-2` 作为默认图像模型，但 chat 前端不应把它当成唯一依赖

参考：

- OpenAI 图像生成指南：<https://platform.openai.com/docs/guides/image-generation>
- OpenAI 图像工具指南：<https://platform.openai.com/docs/guides/tools-image-generation>
- OpenAI 图像编辑接口：<https://platform.openai.com/docs/api-reference/images/createEdit>

## 6. 开源方案选型结论

选型结论已经确定：

- 画布底座采用 `react-konva / Konva`

原因：

- React 集成自然，和当前 Next.js + React + Zustand 架构匹配
- 适合做：
  - 参考图摆放
  - 当前工作图显示
  - 扩图边界控制
  - 后续选区 / mask / 多图输入
- 作为长期图像工作空间底层最稳

其他方案结论：

- `TUI Image Editor`
  - 适合快速做 MVP
  - 不作为当前主线
- `Open WebUI`
  - 适合参考产品形态
  - 不直接嵌入
- `InvokeAI`
  - 适合参考统一画布和图像工作流
  - 不直接嵌入
- `ComfyUI`
  - 适合参考后端工作流
  - 不作为前端嵌入方案

## 7. 与 Mask 的关系

当前产品核心不是先定义 mask，而是先定义工作空间。

因此，第一阶段的顺序应当是：

1. 先做 `Image Studio` 工作空间
2. 再决定 mask 如何作为入口接入

mask 在这个体系里的职责应当是：

- 提供入口
- 提供默认模型
- 提供候选模型范围
- 提供默认参数
- 指示“从这个 mask 创建会话时，推荐进入 image studio”

建议 mask 结构只保留这类字段：

```ts
type Mask = {
  id: string;
  name: string;
  description?: string;
  candidateModels?: ModelCandidate[];
  modelConfig: ModelConfig;

  createAsStudio?: boolean;
  studioKind?: "image";
  imageConfig?: {
    defaultAction?: "auto" | "generate" | "edit";
    defaultSize?: string;
    defaultQuality?: string;
    defaultBackground?: "auto" | "transparent" | "opaque";
    outputFormat?: "png" | "webp" | "jpeg";
    allowReferenceImages?: boolean;
    allowMaskEditing?: boolean;
    allowMultiImageInput?: boolean;
  };
};
```

### 7.1 第一版创建路径约束

第一版明确采用两条独立创建路径：

```ts
createSession("chat", { mask: "xxxxx" })
createSession("studio", { kind: "image" })
```

含义如下：

- `chat` 会话可以继续使用现有 mask 体系
- `studio/image` 工作空间不引入 mask
- 第一版两条创建路径共存，但不交叉

第一版明确不支持：

```ts
createSession("studio", { kind: "image", mask: "xxxxx" })
createSession("chat", { kind: "image" })
```

原因：

- 当前最重要的是先让 `Image Studio` 作为独立工作空间成立
- 如果现在把 mask 引入 `studio`，会同时搅在一起两个问题：
  - 新会话形态如何定义
  - 模板体系如何接入
- 这会让数据结构、优先级规则和 UI 入口同时复杂化

## 8. 当前可用范围

截至 `2026-05-17`，`Image Studio` 已经具备可试用的第一版能力：

- 可以从侧边栏或新建入口创建 `图像工作空间`
- 工作空间有独立状态，不复用普通 chat 消息流
- 可以选择运行时可用的图像模型
- 已支持：
  - 文生图 `generate`
  - 基于参考图的 `edit`
  - 基于编辑链路实现的首版 `expand`
- 可以上传参考图，并在画布中查看当前结果

现阶段产品目标是先“用起来”，而不是继续扩大功能面。

因此，后续实现优先级应调整为：

1. 先验证这条工作流是否顺手、是否满足真实使用场景
2. 再按使用反馈补最必要的交互和接口能力
3. 暂不提前堆积复杂编辑器能力

## 9. 待办

以下事项明确进入待办，不阻塞当前试用：

- `expand` 增加方向控制
  - 例如向左扩、向右扩、向上扩、向下扩
- `edit` 增加选区或 `mask` 输入
- 画布增加更明确的资源切换与预览交互
- `Image Studio` 增加更清晰的结果历史与版本回看
- 梳理 `studio/image` 的专用模型范围规则
  - 不长期复用普通 chat 的通用模型配置
- 评估是否需要把执行层做成可插拔扩展
  - 产品本身仍然保持内建工作空间

阶段性原则：

- `mask` 只服务于 `chat`
- `studio/image` 只服务于图像工作空间
- 后续如果 Image Studio 稳定，再单独讨论是否引入 `studio template`

### 7.2 会话与工作空间的共存方式

当前 Chat 应用不应拆成两个独立产品区，而应共用同一套 session 容器。

建议共存方式：

- 左侧列表统一展示所有 session
- `chat` 与 `studio` 通过图标或类型信号区分
- 新建入口统一分流，而不是分别维护两套系统

建议入口形态：

1. 用户点击 `新建`
2. 选择：
   - `对话`
   - `Image Studio`
3. 创建后进入不同主视图

共存原则：

- 共用同一套会话列表、持久化、同步、搜索、删除、重命名能力
- 不共用同一套主交互界面

具体来说：

- `chat`
  - 主视图是消息流
- `studio/image`
  - 主视图是画布 + 参数 + 版本

### 7.3 第一版不做的交叉能力

为了保证边界清晰，第一版不做以下交叉能力：

- 从 chat 会话中直接“切换成” Image Studio
- 从 Image Studio 直接套用 chat mask
- 让 mask 成为 Image Studio 的创建模板
- 让 Image Studio 继承 chat 的上下文消息结构

这些能力都可以在后续阶段单独讨论，但不进入第一版。

## 8. 第一期 PRD

### 8.1 产品名称

第一期统一使用：

- `Image Studio`

`gpt-image-2` 只是默认模型，不进入产品名。

### 8.2 产品目标

第一期目标是做出一个可连续创作的图像会话。

第一期应满足：

- 用户可以一键进入 `Image Studio`
- 用户可以围绕一张图持续执行多轮操作
- 会话不再只是“发 prompt -> 返回一张图”
- 图像能力由工作空间承载，而不是普通 chat 的附属参数

### 8.3 用户场景

第一期优先支持：

1. 用户输入 prompt 直接生成首张图
2. 用户上传一张参考图，再结合 prompt 生成新图
3. 用户基于已有结果图继续迭代
4. 用户对当前图执行扩图

第一期暂不覆盖：

1. 多图融合
2. 局部涂抹 mask
3. 图层化编辑
4. 批量变体流水线

### 8.4 页面结构

第一期页面采用三栏结构。

左栏：会话与版本区

- 当前会话标题
- 当前工作图列表
- 历史结果图
- 线性版本列表

中栏：Konva 画布区

- 当前工作图
- 扩图边界框
- 参考图预览
- 后续可扩展的选区 / mask 基础层

右栏：参数与操作区

- 模型
- 动作类型
- prompt
- 尺寸 / 比例
- 质量
- 背景
- 输出格式
- 上传参考图 / 设为工作图
- 提交按钮

### 8.5 核心对象

建议第一期明确以下对象。

`Session`

```ts
type StudioSession = {
  id: string;
  topic: string;
  type: "studio";
  studio: {
    kind: "image";
    currentAssetId?: string;
    currentJobDraft?: ImageJobDraft;
  };
};
```

`Asset`

```ts
type ImageAsset = {
  id: string;
  type: "reference" | "generated";
  url: string;
  parentAssetId?: string;
  createdAt: number;
};
```

`Job`

```ts
type ImageAction =
  | "generate"
  | "reference_generate"
  | "edit"
  | "outpaint";

type ImageJob = {
  id: string;
  action: ImageAction;
  prompt: string;
  model: string;
  assetId?: string;
  referenceAssetIds?: string[];
  params: {
    size?: string;
    quality?: string;
    background?: string;
    outputFormat?: string;
  };
  createdAt: number;
};
```

`CanvasState`

```ts
type ImageCanvasState = {
  zoom: number;
  offsetX: number;
  offsetY: number;
  outpaintBounds?: {
    top: number;
    right: number;
    bottom: number;
    left: number;
  };
};
```

### 8.6 第一期动作范围

第一期只开放 4 个动作：

1. `generate`
2. `reference_generate`
3. `edit`
4. `outpaint`

定义如下：

- `generate`
  - 输入文本 prompt
  - 直接生成图像
- `reference_generate`
  - 上传一张参考图
  - 结合 prompt 生成新图
- `edit`
  - 以当前工作图为输入继续迭代
  - 暂不要求局部选区
- `outpaint`
  - 基于当前工作图扩展边界
  - 由画布区提供边界调整

第一期不开放：

- `inpaint`
- `multi_reference`
- `batch_variants`

### 8.7 交互流程

流程 A：首次生成

1. 用户进入 `Image Studio`
2. 右栏默认动作为 `generate`
3. 用户输入 prompt 并选择参数
4. 提交后生成结果图
5. 结果图进入左栏历史与中栏当前工作图

流程 B：参考图生成

1. 用户上传参考图
2. 系统创建 reference asset
3. 用户切换动作为 `reference_generate`
4. 用户输入 prompt
5. 提交后生成结果图

流程 C：继续编辑

1. 用户从历史结果图中选中一张图
2. 将其设为当前工作图
3. 用户切换动作为 `edit`
4. 用户输入新的 prompt
5. 提交后生成新结果图，并记录 parent asset 关系

流程 D：扩图

1. 用户选中当前工作图
2. 用户切换动作为 `outpaint`
3. 中栏画布显示扩图边界控制
4. 用户调整边界后提交
5. 系统生成扩展后的新图

### 8.8 结果图操作

第一期每张结果图至少支持：

- `设为工作图`
- `继续编辑`
- `扩图`
- `下载`

### 8.9 第一版不做项

第一版明确不做：

- 局部重绘 `inpaint`
- 多图融合
- 图层系统
- 资产库首页
- 批量生成工作流
- 风格模板市场
- 独立图像项目空间

### 8.10 验收标准

第一期完成后，应至少满足以下验收条件：

1. 用户可以从入口创建 `Image Studio` 会话。
2. `Image Studio` 会话的数据结构包含：
   - `type = "studio"`
   - `studio.kind = "image"`
3. 会话中模型列表只显示运行时支持图像能力的模型。
4. 用户可以完成一次纯文生图操作，并看到结果图。
5. 用户可以上传参考图并完成一次 `reference_generate`。
6. 用户可以将历史结果图设为当前工作图，再执行一次 `edit`。
7. 用户可以在画布中调整扩图边界并执行一次 `outpaint`。
8. 每张结果图至少支持：
   - `设为工作图`
   - `继续编辑`
   - `扩图`
   - `下载`
9. 结果图历史可以在会话内被重新选中与复用。
10. 第一版无需支持局部重绘和多图融合。

## 9. 后续演进

建议分三期推进。

第一期：

- 建立 `Image Studio` 工作空间
- 打通 4 个核心动作
- 建立版本复用链路

第二期：

- 增加 `inpaint`
- 增加局部选区 / brush / mask
- 增加多图参考

第三期：

- 增加图像资产管理
- 增加批量工作流
- 增加更复杂的图像项目空间能力

## 10. 建议的实现改造点

- `app/store/chat.ts`
  - 增加 `SessionType`
  - 增加 `studio.kind`
- `app/store/mask.ts`
  - 增加 `createAsStudio`
  - 增加 `studioKind`
  - 增加 `imageConfig`
- `app/components/chat.tsx`
  - 对 `session.type = studio && session.studio.kind = image` 渲染图像工作空间
- `app/client/api.ts`
  - 增加图像能力辅助判断
- `app/client/platforms/openai.ts`
  - 拆分图像生成与后续编辑动作
- `app/utils.ts`
  - 弱化 `dall-e-3` 模型名硬编码

## 11. 当前结论

当前结论已经明确：

- 产品名称：`Image Studio`
- 会话结构：`SessionType = "chat" | "studio"` + `StudioKind = "image"`
- 技术底座：`react-konva / Konva`
- 第一版目标：连续图像创作会话
- 第一版动作：`generate`、`reference_generate`、`edit`、`outpaint`

后续图像能力相关讨论以本文为主，不再并行维护多份重复文档。
