# 技能发布、上线、配置与使用流程

本文档服从 [AI Native 能力分层架构](./AI-Native能力分层架构.md)。后续统一按“技能、工具、模型、存储”四层理解发布和运行流程；旧的插件等说法只作为技术兼容概念，不作为用户侧一级产品概念。

本文说明技能和工具从编写、发布、上线、配置到用户使用的完整流程。当前工具主要通过 MCP Server 承载。重点解决五个问题：

- 开发者怎么添加一个新技能
- 开发者怎么添加一个新工具（MCP Server）
- 维护者怎么审核并上线
- Chat 应用怎么发现和安装技能、工具
- 用户怎么配置并使用技能

## 1. 核心结论

技能不是一个提示词文件，也不是工具本身。技能是一个可被发现、安装、配置和运行的任务入口。

当前推荐流程：

1. 技能作者在社区市场仓库提交技能包。
2. 社区市场仓库生成并提交技能发布清单。
3. Chat 发现页加载远程 `packages.json`。
4. 用户在发现页安装技能。
5. Chat 把技能包转换成本地技能配置。
6. 用户按提示补齐模型、工具或存储配置。
7. 用户点击技能进入对应会话或工作区。

工具推荐流程：

1. 工具作者在社区市场仓库提交工具包，当前主要由 MCP Server 承载。
2. 社区市场仓库生成并提交工具发布清单。
3. Chat 发现页和工具配置页按多源策略加载 `mcp/packages.json`。
4. 用户安装并配置工具。
5. 依赖该工具的技能状态变为可用。

## 2. 仓库分工

### Chat 仓库

仓库地址：`https://github.com/yeying-community/chat`

Chat 仓库负责：

- 内置技能定义：`app/skills/cn.ts`、`app/skills/en.ts`
- 内置技能包生成：`public/skill-packages.json`
- 发现页展示：`app/components/discovery.tsx`
- 工具配置页：`app/components/mcp-market.tsx`
- 技能安装转换：`app/skills/package.ts`
- 技能运行状态判断：`app/skills/runtime.ts`
- 技能本地存储：`app/store/skill.ts`

Chat 仓库不应该长期维护社区市场内容。社区技能、工具、模型服务商、存储能力等可扩展内容应该进入社区市场仓库。

### 社区市场仓库

仓库地址：`https://github.com/yeying-community/marketplace`

当前社区市场仓库负责：

- 社区技能源码：`skills/cn/*.json`、`skills/en/*.json`
- 社区工具源码：`mcp/servers/*.json`
- 技能包模板：`templates/skill.json`
- 工具包模板：`templates/mcp-server.json`
- 技能包校验与生成：`scripts/build.mjs`
- 技能发布清单：`packages.json`
- 技能轻量索引：`index.json`
- 工具发布清单：`mcp/packages.json`
- 工具轻量索引：`mcp/index.json`

后续可以扩展：

- 模型服务商清单：`providers/packages.json`
- 存储能力清单：`storage/packages.json`

不建议再新增用户侧“插件市场”或“工具市场”。工具作为底层能力被技能依赖，模型和存储作为独立能力层被技能声明和运行检查。

Chat 默认按多源顺序加载技能市场：

```txt
1. https://raw.githubusercontent.com/yeying-community/marketplace/main/packages.json
2. https://cdn.jsdelivr.net/gh/yeying-community/marketplace@main/packages.json
3. /marketplace/packages.json
```

Chat 默认按多源顺序加载工具市场。当前实现路径仍是 `mcp/packages.json`：

```txt
1. https://raw.githubusercontent.com/yeying-community/marketplace/main/mcp/packages.json
2. https://cdn.jsdelivr.net/gh/yeying-community/marketplace@main/mcp/packages.json
3. /marketplace/mcp/packages.json
```

其中 `/marketplace/*.json` 是 Chat 仓库内置的市场快照，用于 GitHub raw 和 CDN 都不可用时兜底。生产环境不应该因为 jsDelivr 缓存滞后或 GitHub 网络波动导致发现页完全不可用。

本地开发新技能或工具时，不依赖远程 GitHub 和 CDN，推荐运行：

```bash
npm run dev:marketplace
```

该命令会启动本地 marketplace 静态服务，并把 Chat 的市场源指向本地清单。发现页顶部会展示实际命中的市场源、技能数量和工具数量。

如果需要更新 Chat 内置市场快照，运行：

```bash
npm run marketplace:snapshot
```

该命令会从相邻的 `marketplace` 仓库复制 `packages.json` 和 `mcp/packages.json` 到 Chat 的 `public/marketplace`。内置快照只用于远程源不可用时兜底，不是社区市场上线的主链路。

## 3. 技能包是什么

一个技能包是一个 JSON manifest，描述这个技能如何被发现、安装、配置和运行。

最小结构：

```json
{
  "schemaVersion": "1.0",
  "id": "web-research",
  "version": "1.0.0",
  "name": {
    "cn": "网页调研",
    "en": "Web Research"
  },
  "description": {
    "cn": "围绕一个问题搜索资料、提炼来源、输出结构化结论。"
  },
  "category": "research",
  "launch": {
    "type": "chat"
  },
  "instructions": {
    "type": "inline",
    "content": "你是网页调研助手..."
  },
  "model": {
    "syncGlobalConfig": true,
    "candidates": []
  },
  "tools": [],
  "mcp": {
    "servers": []
  },
  "permissions": {
    "network": false,
    "filesystem": false,
    "wallet": false,
    "externalTools": []
  },
  "release": {
    "status": "published",
    "review": "approved"
  }
}
```

字段规范详见：

- Chat 仓库：`docs/技能规范.md`
- 社区市场仓库：`docs/SKILL_SPEC.md`

## 4. 技能发布流程

### 4.1 作者添加技能

在社区市场仓库操作：

```bash
git clone https://github.com/yeying-community/marketplace.git
cd marketplace
cp templates/skill.json skills/cn/<skill-id>.json
```

然后编辑新文件，重点填写：

- `id`：稳定 ID，使用 kebab-case，例如 `reading-summary`
- `name`：用户看到的技能名称，必须清晰聚焦
- `description`：告诉用户什么时候使用
- `launch`：点击后进入哪里
- `instructions`：模型运行时的任务说明
- `model`：默认模型、候选模型或是否跟随全局模型
- `tools`：工具依赖，例如模型内置 `web_search`
- `mcp.servers`：由 MCP Server 承载的工具依赖，例如 `fetch`、`brave-search`
- `storage`：后续可扩展的存储或知识库依赖
- `permissions`：网络、文件、钱包等权限声明
- `release`：发布状态和审核状态

### 4.2 生成发布清单

在社区市场仓库运行：

```bash
npm run check
```

这个命令会：

- 校验 `skills/cn/*.json` 和 `skills/en/*.json`
- 生成 `index.json`
- 生成 `packages.json`

`packages.json` 是 Chat 当前实际加载的远程技能包列表，必须提交到 GitHub。

### 4.3 提交 PR

PR 应包含：

- 新增或修改的技能 JSON
- 更新后的 `index.json`
- 更新后的 `packages.json`

PR 描述建议说明：

- 技能解决什么任务
- 需要什么模型能力
- 需要哪些工具，以及这些工具来自模型内置能力还是工具运行时
- 是否有必需配置
- 用户点击后进入聊天还是专属工作区

## 5. 工具发布流程

### 5.1 作者添加工具

在社区市场仓库操作：

```bash
git clone https://github.com/yeying-community/marketplace.git
cd marketplace
cp templates/mcp-server.json mcp/servers/<mcp-id>.json
```

然后编辑新文件，重点填写：

- `id`：稳定 ID，使用 kebab-case，例如 `brave-search`
- `name`：用户看到的工具名称，必须清晰聚焦
- `description`：告诉用户这个工具提供什么模型可调用能力
- `repo`：源码或官方说明地址
- `command`：启动命令，例如 `npx`、`uvx`
- `baseArgs`：启动参数，不包含用户密钥
- `configurable`：是否需要用户配置
- `configSchema`：需要用户填写的配置项
- `argsMapping`：配置项如何映射到参数或环境变量
- `release`：发布状态和审核状态

### 5.2 生成工具发布清单

在社区市场仓库运行：

```bash
npm run check
```

这个命令会：

- 校验 `mcp/servers/*.json`
- 生成 `mcp/index.json`
- 生成 `mcp/packages.json`

`mcp/packages.json` 是 Chat 当前实际加载的社区工具包列表，必须提交到 GitHub。当前这些工具包由 MCP Server 承载。Chat 会先尝试 GitHub raw，再尝试 CDN，最后回落到 Chat 内置快照。

### 5.3 提交 PR

PR 应包含：

- 新增或修改的工具 JSON
- 更新后的 `mcp/index.json`
- 更新后的 `mcp/packages.json`

PR 描述建议说明：

- 工具提供什么模型可调用能力
- 启动命令来自哪里
- 是否需要密钥或本地权限
- 可能被哪些技能依赖

## 6. 上线流程

### 6.1 维护者审核

维护者审核重点：

- 名称是否清晰，不能是“高效助手”这类泛化名称
- 描述是否能让用户判断使用场景
- instructions 是否服务于明确任务
- 是否包含密钥、私有 URL、本地路径
- 工具、模型和存储依赖是否显式声明
- 必需依赖是否标记为 `required: true`
- 可选依赖是否标记为 `required: false`
- 模型候选是否合理，是否过度绑定单一不可用模型
- 权限声明是否与任务一致

工具额外审核：

- 启动命令是否来自可信包或可信仓库
- 是否要求用户密钥，且密钥只通过配置项注入
- 是否包含本地绝对路径、私有地址或硬编码 token
- 权限边界是否清晰，例如文件、网络、钱包
- 是否能被技能通过稳定工具 ID 依赖

### 6.2 合并后上线

PR 合并到社区市场仓库 `main` 后，技能上线通常不需要 Chat 发版。

原因：

- Chat 发现页会按 `GitHub raw -> jsDelivr CDN -> 内置快照` 加载 `packages.json`
- Chat 发现页和工具配置页会按 `GitHub raw -> jsDelivr CDN -> 内置快照` 加载 `mcp/packages.json`
- 新技能或工具合并后，远程源更新后会出现在发现页
- 如果 CDN 缓存滞后，Chat 会继续尝试 GitHub raw；如果远程都不可用，则显示内置快照，避免发现页不可用

注意：

- 如果 Chat 代码不支持某种新字段，字段只能作为元数据展示，不应影响旧版本运行。
- 如果技能依赖新的运行时能力，需要先升级 Chat，再发布技能。
- 如果内置快照需要包含最新市场内容，仍然需要更新 Chat 仓库的 `public/marketplace` 快照并发版；这只是远程不可用时的兜底，不是上线主链路。

## 7. Chat 如何发现和安装

### 7.1 发现

Chat 发现页加载两类技能：

- 本地内置技能包列表：`/skill-packages.json`
- 社区技能包列表：`GitHub raw -> jsDelivr CDN -> /marketplace/packages.json`
- 社区工具包列表：`GitHub raw -> jsDelivr CDN -> /marketplace/mcp/packages.json`

展示时统一归类为“技能”，区别只在来源：

- 官方：Chat 内置技能
- 社区：社区市场仓库发布的技能
- 自定义：用户本地创建的技能

工具展示时统一归类为“工具”，来源分为：

- 官方：Chat 内置工具预设
- 社区：社区市场仓库发布的工具

同 ID 的工具同时存在时，Chat 优先使用官方内置预设，避免远程配置覆盖核心能力。

### 7.2 安装技能

用户点击“安装”后，Chat 会：

1. 读取社区 `SkillPackage`
2. 调用 `skillPackageToSkill()`
3. 写入 `useSkillStore`
4. 记录 `packageId`
5. 解析运行状态

安装后的技能进入用户本地技能库。之后即使远程技能包更新，已安装技能也不会自动改变当前本地配置。

### 7.3 安装工具

用户点击工具后，Chat 会进入工具配置页：

1. 加载内置工具预设和社区 `mcp/packages.json`
2. 按工具 ID 合并列表
3. 用户点击添加
4. 如果工具需要配置，先填写配置项
5. 写入本地工具运行配置
6. 工具客户端启动后，依赖它的技能可进入 `ready`

### 7.4 运行状态

Chat 会把技能解析成三个状态：

| 状态           | 含义                         | 用户动作             |
| -------------- | ---------------------------- | -------------------- |
| `ready`        | 模型和必需工具都可用       | 可以直接开始使用       |
| `needs_config` | 有依赖缺失，但可以通过配置补齐 | 去技能或工具配置页配置 |
| `unavailable`  | 当前运行环境无法满足技能要求 | 更换模型或等待能力上线 |

## 8. 配置流程

技能配置分三类：模型、工具、存储。模型服务商可能提供内置能力，例如 `web_search`。MCP 是当前工具层的主要运行时实现，也有密钥、路径、远程地址等运行配置。

### 8.1 模型配置

技能可以选择：

- 跟随全局模型：`syncGlobalConfig: true`
- 使用技能默认模型：`syncGlobalConfig: false`
- 限制候选模型：`candidates`

原则：

- 普通聊天技能默认跟随全局模型。
- 图片、语音、搜索、代码等专业技能可以限制候选模型。
- OpenAI 兼容模型优先面向 `/v1/responses`。
- Anthropic 模型优先面向 `/v1/messages`。
- `/v1/chat/completions` 只作为旧服务兼容路径。

### 8.2 模型内置能力配置

模型内置能力是模型服务商直接提供的能力，例如：

```json
{
  "id": "web_search",
  "name": "Web Search",
  "required": false
}
```

这类能力不需要用户安装 MCP，但需要当前模型和 provider 支持。

### 8.3 工具配置

工具是模型可调用的外部能力。当前主要通过 MCP Server 承载，例如：

```json
{
  "id": "fetch",
  "name": "Fetch",
  "transport": "stdio",
  "required": false
}
```

配置流程：

1. 用户进入发现页或工具配置页。
2. 找到技能提示缺失的工具。
3. 安装或启用对应工具服务。
4. 回到技能页面确认状态变为 `ready` 或 `needs_config`。

如果工具是可选依赖，缺失时不应阻止使用，只能影响效果。

如果工具是必需依赖，缺失时技能不应直接运行，应提示用户先配置。

### 8.4 存储配置

存储用于保存应用数据、文件、知识库和同步状态。默认情况下，存储不等于 MCP。

当技能需要让模型读取或写入存储数据时，应通过工具暴露，例如文件库 MCP、知识库 MCP 或云盘 MCP。

## 9. 使用流程

### 9.1 普通聊天

用户不选择任何技能时：

- 默认进入普通对话
- 使用当前全局模型
- 不注入技能 instructions
- 不强制绑定工具

这是基础会话能力，不能被技能体系复杂化。

### 9.2 使用聊天类技能

用户点击聊天类技能后：

1. Chat 创建新会话
2. 会话绑定该技能
3. 注入技能 instructions
4. 应用技能模型配置
5. 只注入该技能声明的工具和 MCP
6. 用户直接输入任务

示例：

- 网页调研
- 阅读总结
- 方案对比
- 代码审查

### 9.3 使用工作区类技能

工作区类技能不一定进入聊天页面。

例如图片创作：

```json
{
  "launch": {
    "type": "workspace",
    "target": "sd"
  }
}
```

用户点击后应进入图片创作页面，而不是普通聊天会话。

原则：

- 技能入口服务于任务
- 任务需要专属 UI 时进入专属工作区
- 不要让用户先选“类型”再选技能

## 10. 升级、下架和兼容

### 10.1 升级

远程技能包升级时：

- 新用户安装时使用新版本
- 已安装技能不应被静默覆盖
- 历史会话不应被远程更新改变行为

后续可以增加“发现新版本，用户确认升级”的流程。

### 10.2 下架

技能不推荐直接从历史清单中硬删除。

建议：

```json
{
  "release": {
    "status": "deprecated"
  }
}
```

处理原则：

- `deprecated`：发现页不推荐，但已安装用户可以继续使用
- `removed`：不应再提交到发布清单
- 已安装技能是否删除，由用户或管理员决定

### 10.3 兼容

如果技能依赖 Chat 新能力，应在 `compatibility` 里声明：

```json
{
  "compatibility": {
    "appVersion": ">=0.1.0",
    "mcp": "optional"
  }
}
```

Chat 旧版本不理解的新字段，不能导致崩溃。

## 11. 常见问题

### 市场拿到的是直接可用技能，还是模板？

应该是“可安装的技能包”，不是单纯模板。

但技能是否能立即运行，取决于用户环境：

- 模型是否可用
- 工具是否已安装或启用
- 必需权限是否满足
- 当前 Chat 版本是否支持该技能声明的能力

所以市场技能的正确状态不是“直接能跑或不能跑”，而是：

- 可安装
- 已安装
- 可配置
- 可运行
- 不可用

### 为什么技能要声明工具，而不是直接包含工具实现？

技能只声明任务和依赖。工具实现是运行时服务，需要用户本地或组织环境提供。当前工具实现主要通过 MCP Server 承载。

这样做的好处：

- 技能可以跨环境发布
- 工具可以独立升级
- 用户能看清楚技能需要什么工具
- 不把密钥、服务地址和本地路径写进技能包

### OpenAPI 导入和工具是什么关系？

OpenAPI 导入只是 HTTP API 描述。它不应该作为和工具平级的产品主入口。

更合理的路径：

1. 用户或开发者导入 OpenAPI 描述
2. 系统适配成工具
3. 技能声明依赖该工具
4. 模型通过 MCP 调用工具

## 12. 推荐落地顺序

短期：

- 用社区市场仓库维护社区技能包
- Chat 发现页加载远程 `packages.json`
- 用户安装后本地配置模型、工具和存储
- 缺失依赖显示 `needs_config`

中期：

- 支持技能版本升级提示
