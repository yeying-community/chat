# 技能发布、上线、配置与使用流程

本文说明技能和 MCP 从编写、发布、上线、配置到用户使用的完整流程。重点解决五个问题：

- 开发者怎么添加一个新技能
- 开发者怎么添加一个新 MCP
- 维护者怎么审核并上线
- Chat 应用怎么发现和安装技能、MCP
- 用户怎么配置并使用技能

## 1. 核心结论

技能不是一个提示词文件，也不是 MCP 本身。技能是一个可被发现、安装、配置和运行的任务入口。

当前推荐流程：

1. 技能作者在社区市场仓库提交技能包。
2. 社区市场仓库生成并提交技能发布清单。
3. Chat 发现页加载远程 `packages.json`。
4. 用户在发现页安装技能。
5. Chat 把技能包转换成本地技能配置。
6. 用户按提示补齐模型、MCP 或工具配置。
7. 用户点击技能进入对应会话或工作区。

MCP 推荐流程：

1. MCP 作者在社区市场仓库提交 MCP 服务器包。
2. 社区市场仓库生成并提交 MCP 发布清单。
3. Chat 发现页和 MCP 页面加载远程 `mcp/packages.json`。
4. 用户安装并配置 MCP。
5. 依赖该 MCP 的技能状态变为可用。

## 2. 仓库分工

### Chat 仓库

仓库地址：`https://github.com/yeying-community/chat`

Chat 仓库负责：

- 内置技能定义：`app/skills/cn.ts`、`app/skills/en.ts`
- 内置技能包生成：`public/skill-packages.json`
- 发现页展示：`app/components/discovery.tsx`
- MCP 市场页：`app/components/mcp-market.tsx`
- 技能安装转换：`app/skills/package.ts`
- 技能运行状态判断：`app/skills/runtime.ts`
- 技能本地存储：`app/store/skill.ts`

Chat 仓库不应该长期维护社区市场内容。社区技能、MCP、模型服务商等可扩展内容应该进入社区市场仓库。

### 社区市场仓库

仓库地址：`https://github.com/yeying-community/marketplace`

当前社区市场仓库负责：

- 社区技能源码：`skills/cn/*.json`、`skills/en/*.json`
- 社区 MCP 源码：`mcp/servers/*.json`
- 技能包模板：`templates/skill.json`
- MCP 包模板：`templates/mcp-server.json`
- 技能包校验与生成：`scripts/build.mjs`
- 技能发布清单：`packages.json`
- 技能轻量索引：`index.json`
- MCP 发布清单：`mcp/packages.json`
- MCP 轻量索引：`mcp/index.json`

后续可以扩展：

- 模型服务商清单：`providers/packages.json`
- 工具适配包清单：`tools/packages.json`

Chat 远程加载地址：

```txt
https://raw.githubusercontent.com/yeying-community/marketplace/main/packages.json
```

MCP 远程加载地址：

```txt
https://raw.githubusercontent.com/yeying-community/marketplace/main/mcp/packages.json
```

## 3. 技能包是什么

一个技能包是一个 JSON manifest，描述这个技能如何被发现、安装、配置和运行。

最小结构：

```json
{
  "schemaVersion": "1.0",
  "id": "web-research",
  "version": "1.0.0",
  "name": {
    "cn": "网页调研",
    "en": "Web Research"
  },
  "description": {
    "cn": "围绕一个问题搜索资料、提炼来源、输出结构化结论。"
  },
  "category": "research",
  "launch": {
    "type": "chat"
  },
  "instructions": {
    "type": "inline",
    "content": "你是网页调研助手..."
  },
  "model": {
    "syncGlobalConfig": true,
    "candidates": []
  },
  "tools": [],
  "mcp": {
    "servers": []
  },
  "permissions": {
    "network": false,
    "filesystem": false,
    "wallet": false,
    "externalTools": []
  },
  "release": {
    "status": "published",
    "review": "approved"
  }
}
```

字段规范详见：

- Chat 仓库：`docs/技能规范.md`
- 社区市场仓库：`docs/SKILL_SPEC.md`

## 4. 技能发布流程

### 4.1 作者添加技能

在社区市场仓库操作：

```bash
git clone https://github.com/yeying-community/marketplace.git
cd marketplace
cp templates/skill.json skills/cn/<skill-id>.json
```

然后编辑新文件，重点填写：

- `id`：稳定 ID，使用 kebab-case，例如 `reading-summary`
- `name`：用户看到的技能名称，必须清晰聚焦
- `description`：告诉用户什么时候使用
- `launch`：点击后进入哪里
- `instructions`：模型运行时的任务说明
- `model`：默认模型、候选模型或是否跟随全局模型
- `tools`：内置工具，例如 `web_search`
- `mcp.servers`：MCP 依赖，例如 `fetch`、`brave-search`
- `permissions`：网络、文件、钱包等权限声明
- `release`：发布状态和审核状态

### 4.2 生成发布清单

在社区市场仓库运行：

```bash
npm run check
```

这个命令会：

- 校验 `skills/cn/*.json` 和 `skills/en/*.json`
- 生成 `index.json`
- 生成 `packages.json`

`packages.json` 是 Chat 当前实际加载的远程技能包列表，必须提交到 GitHub。

### 4.3 提交 PR

PR 应包含：

- 新增或修改的技能 JSON
- 更新后的 `index.json`
- 更新后的 `packages.json`

PR 描述建议说明：

- 技能解决什么任务
- 需要什么模型能力
- 需要哪些 MCP 或内置工具
- 是否有必需配置
- 用户点击后进入聊天还是专属工作区

## 5. MCP 发布流程

### 5.1 作者添加 MCP

在社区市场仓库操作：

```bash
git clone https://github.com/yeying-community/marketplace.git
cd marketplace
cp templates/mcp-server.json mcp/servers/<mcp-id>.json
```

然后编辑新文件，重点填写：

- `id`：稳定 ID，使用 kebab-case，例如 `brave-search`
- `name`：用户看到的 MCP 名称，必须清晰聚焦
- `description`：告诉用户这个 MCP 提供什么工具能力
- `repo`：源码或官方说明地址
- `command`：启动命令，例如 `npx`、`uvx`
- `baseArgs`：启动参数，不包含用户密钥
- `configurable`：是否需要用户配置
- `configSchema`：需要用户填写的配置项
- `argsMapping`：配置项如何映射到参数或环境变量
- `release`：发布状态和审核状态

### 5.2 生成 MCP 发布清单

在社区市场仓库运行：

```bash
npm run check
```

这个命令会：

- 校验 `mcp/servers/*.json`
- 生成 `mcp/index.json`
- 生成 `mcp/packages.json`

`mcp/packages.json` 是 Chat 当前实际加载的远程 MCP 服务器包列表，必须提交到 GitHub。

### 5.3 提交 PR

PR 应包含：

- 新增或修改的 MCP JSON
- 更新后的 `mcp/index.json`
- 更新后的 `mcp/packages.json`

PR 描述建议说明：

- MCP 提供什么工具能力
- 启动命令来自哪里
- 是否需要密钥或本地权限
- 可能被哪些技能依赖

## 6. 上线流程

### 6.1 维护者审核

维护者审核重点：

- 名称是否清晰，不能是“高效助手”这类泛化名称
- 描述是否能让用户判断使用场景
- instructions 是否服务于明确任务
- 是否包含密钥、私有 URL、本地路径
- MCP 和工具依赖是否显式声明
- 必需依赖是否标记为 `required: true`
- 可选依赖是否标记为 `required: false`
- 模型候选是否合理，是否过度绑定单一不可用模型
- 权限声明是否与任务一致

MCP 额外审核：

- 启动命令是否来自可信包或可信仓库
- 是否要求用户密钥，且密钥只通过配置项注入
- 是否包含本地绝对路径、私有地址或硬编码 token
- 权限边界是否清晰，例如文件、网络、钱包
- 是否能被技能通过稳定 MCP ID 依赖

### 6.2 合并后上线

PR 合并到社区市场仓库 `main` 后，技能上线不需要 Chat 发版。

原因：

- Chat 发现页会读取远程 `packages.json`
- Chat 发现页和 MCP 页面会读取远程 `mcp/packages.json`
- `packages.json` 位于 GitHub raw 地址
- 新技能或 MCP 合并后会自然出现在发现页

注意：

- 如果 Chat 代码不支持某种新字段，字段只能作为元数据展示，不应影响旧版本运行。
- 如果技能依赖新的运行时能力，需要先升级 Chat，再发布技能。

## 7. Chat 如何发现和安装

### 7.1 发现

Chat 发现页加载两类技能：

- 本地内置技能包列表：`/skill-packages.json`
- 远程社区技能包列表：`https://raw.githubusercontent.com/yeying-community/marketplace/main/packages.json`
- 远程社区 MCP 服务器包列表：`https://raw.githubusercontent.com/yeying-community/marketplace/main/mcp/packages.json`

展示时统一归类为“技能”，区别只在来源：

- 官方：Chat 内置技能
- 社区：社区市场仓库发布的技能
- 自定义：用户本地创建的技能

MCP 展示时统一归类为“MCP”，来源分为：

- 官方：Chat 内置 MCP 预设
- 社区：社区市场仓库发布的 MCP

同 ID 的 MCP 同时存在时，Chat 优先使用官方内置预设，避免远程配置覆盖核心能力。

### 7.2 安装技能

用户点击“安装”后，Chat 会：

1. 读取远程 `SkillPackage`
2. 调用 `skillPackageToSkill()`
3. 写入 `useSkillStore`
4. 记录 `packageId`
5. 解析运行状态

安装后的技能进入用户本地技能库。之后即使远程技能包更新，已安装技能也不会自动改变当前本地配置。

### 7.3 安装 MCP

用户点击 MCP 后，Chat 会进入 MCP 页面：

1. 加载内置 MCP 预设和远程 `mcp/packages.json`
2. 按 MCP ID 合并列表
3. 用户点击添加
4. 如果 MCP 需要配置，先填写配置项
5. 写入本地 MCP 配置
6. MCP 客户端启动后，依赖它的技能可进入 `ready`

### 7.4 运行状态

Chat 会把技能解析成三个状态：

| 状态           | 含义                         | 用户动作             |
| -------------- | ---------------------------- | -------------------- |
| `ready`        | 模型和必需工具都可用         | 可以直接开始使用     |
| `needs_config` | 有依赖缺失，但可以通过配置补齐 | 去技能或 MCP 页面配置 |
| `unavailable`  | 当前运行环境无法满足技能要求 | 更换模型或等待能力上线 |

## 8. 配置流程

技能配置分三类：模型、内置工具、MCP。MCP 本身也有运行配置，例如密钥、路径、远程地址。

### 8.1 模型配置

技能可以选择：

- 跟随全局模型：`syncGlobalConfig: true`
- 使用技能默认模型：`syncGlobalConfig: false`
- 限制候选模型：`candidates`

原则：

- 普通聊天技能默认跟随全局模型。
- 图片、语音、搜索、代码等专业技能可以限制候选模型。
- OpenAI 兼容模型优先面向 `/v1/responses`。
- Anthropic 模型优先面向 `/v1/messages`。
- `/v1/chat/completions` 只作为旧服务兼容路径。

### 7.2 内置工具配置

内置工具是模型服务商直接提供的能力，例如：

```json
{
  "id": "web_search",
  "name": "Web Search",
  "required": false
}
```

这类工具不需要用户安装 MCP，但需要当前模型和 provider 支持。

### 7.3 MCP 配置

MCP 是外部工具运行时，例如：

```json
{
  "id": "fetch",
  "name": "Fetch",
  "transport": "stdio",
  "required": false
}
```

配置流程：

1. 用户进入发现页或 MCP 页面。
2. 找到技能提示缺失的 MCP。
3. 安装或启用对应 MCP 服务。
4. 回到技能页面确认状态变为 `ready` 或 `needs_config`。

如果 MCP 是可选依赖，缺失时不应阻止使用，只能影响效果。

如果 MCP 是必需依赖，缺失时技能不应直接运行，应提示用户先配置。

## 8. 使用流程

### 8.1 普通聊天

用户不选择任何技能时：

- 默认进入普通对话
- 使用当前全局模型
- 不注入技能 instructions
- 不强制绑定 MCP 或工具

这是基础会话能力，不能被技能体系复杂化。

### 8.2 使用聊天类技能

用户点击聊天类技能后：

1. Chat 创建新会话
2. 会话绑定该技能
3. 注入技能 instructions
4. 应用技能模型配置
5. 只注入该技能声明的工具和 MCP
6. 用户直接输入任务

示例：

- 网页调研
- 阅读总结
- 方案对比
- 代码审查

### 8.3 使用工作区类技能

工作区类技能不一定进入聊天页面。

例如图片创作：

```json
{
  "launch": {
    "type": "workspace",
    "target": "sd"
  }
}
```

用户点击后应进入图片创作页面，而不是普通聊天会话。

原则：

- 技能入口服务于任务
- 任务需要专属 UI 时进入专属工作区
- 不要让用户先选“类型”再选技能

## 9. 升级、下架和兼容

### 9.1 升级

远程技能包升级时：

- 新用户安装时使用新版本
- 已安装技能不应被静默覆盖
- 历史会话不应被远程更新改变行为

后续可以增加“发现新版本，用户确认升级”的流程。

### 9.2 下架

技能不推荐直接从历史清单中硬删除。

建议：

```json
{
  "release": {
    "status": "deprecated"
  }
}
```

处理原则：

- `deprecated`：发现页不推荐，但已安装用户可以继续使用
- `removed`：不应再提交到发布清单
- 已安装技能是否删除，由用户或管理员决定

### 9.3 兼容

如果技能依赖 Chat 新能力，应在 `compatibility` 里声明：

```json
{
  "compatibility": {
    "appVersion": ">=0.1.0",
    "mcp": "optional"
  }
}
```

Chat 旧版本不理解的新字段，不能导致崩溃。

## 10. 常见问题

### 市场拿到的是直接可用技能，还是模板？

应该是“可安装的技能包”，不是单纯模板。

但技能是否能立即运行，取决于用户环境：

- 模型是否可用
- MCP 是否已安装或启用
- 必需权限是否满足
- 当前 Chat 版本是否支持该技能声明的能力

所以市场技能的正确状态不是“直接能跑或不能跑”，而是：

- 可安装
- 已安装
- 可配置
- 可运行
- 不可用

### 为什么技能要声明 MCP，而不是直接包含 MCP？

技能只声明任务和依赖。MCP 是运行时服务，需要用户本地或组织环境提供。

这样做的好处：

- 技能可以跨环境发布
- MCP 可以独立升级
- 用户能看清楚技能需要什么工具
- 不把密钥、服务地址和本地路径写进技能包

### OpenAPI 导入和 MCP 是什么关系？

OpenAPI 导入只是 HTTP API 描述。它不应该作为和 MCP 平级的产品主入口。

更合理的路径：

1. 用户或开发者导入 OpenAPI 描述
2. 系统适配成 MCP 工具
3. 技能声明依赖该 MCP 工具
4. 模型通过 MCP 调用工具

## 11. 推荐落地顺序

短期：

- 用社区市场仓库维护社区技能包
- Chat 发现页加载远程 `packages.json`
- 用户安装后本地配置模型、内置工具和 MCP
- 缺失依赖显示 `needs_config`

中期：

- 支持技能版本升级提示
- 支持组织可见范围和员工可见范围
- 支持 MCP 缺失时一键安装或引导配置
- 支持技能详情页展示模型、MCP、权限和兼容性

长期：

- 支持技能审核工作流
- 支持技能评分、下载量和维护状态
- 支持组织私有技能市场
- 支持技能包目录结构和外部 instructions 文件
